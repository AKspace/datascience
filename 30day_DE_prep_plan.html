Week 1 · Days 1–7 Python for Data + PySpark Foundations Core tooling
DAY 01 Learn
Python for Data — Pandas Basics
Load a CSV of sales data. Filter rows, group by category, calculate total revenue per category. Understand DataFrame vs Series.
Output: Jupyter notebook with 10 Pandas operations on real data
DAY 02 Anomaly
Data Quality — Nulls, Duplicates, Type Mismatches
Take a messy dataset (download from Kaggle). Find nulls, duplicates, wrong data types. Fix each one. This is what DE work actually looks like daily.
Output: Clean dataset + documented data quality issues found
DAY 03 Learn
PySpark — SparkSession, RDDs, DataFrames
Start PySpark locally. Create a SparkSession. Load a CSV into a Spark DataFrame. Run filter(), select(), groupBy(), agg(). Compare to Pandas — feel the difference.
Output: First PySpark script that reads, filters and aggregates data
DAY 04 Anomaly
PySpark — Lazy Evaluation in Action
Write 5 transformations on a large file. Add .explain() to see the execution plan. Observe that nothing runs until .show() or .count(). Break a job intentionally — fix it.
Output: Annotated script showing DAG + lazy evaluation in action
DAY 05 Learn
PySpark — Joins & Window Functions
Join two DataFrames (customers + orders). Then use Window functions: RANK, ROW_NUMBER, LAG on real data. This directly maps to SQL knowledge you now have.
Output: PySpark script with 3 join types + 3 window function examples
DAY 06 Anomaly
Shuffle Problem — Why Your Job is Slow
Run a groupBy on a large dataset. Check Spark UI (localhost:4040). See the shuffle read/write. Then repartition() before groupBy and compare job time. This is a real production scenario.
Output: Before/after timing comparison with explanation of shuffle cost
DAY 07 Build
Mini Project — Sales Analytics Pipeline
Build end-to-end: ingest raw CSV → clean data → transform with PySpark → answer 5 business questions → write output as Parquet. Push to GitHub.
Output: GitHub repo — Project 1 of 3 complete ✅
Project 1 Sales Analytics Pipeline — End to End

Build a pipeline that ingests raw retail sales CSV data, cleans it (handle nulls, fix types, remove duplicates), transforms it using PySpark (aggregations, window functions), answers 5 business questions, and writes the output as Parquet files. This is a real DE deliverable, not a tutorial exercise.

  • 1
    Download a real dataset from Kaggle (retail sales or e-commerce)
  • 2
    Ingest with PySpark, profile the data — find all quality issues anomaly
  • 3
    Write cleaning functions — handle nulls, cast types, deduplicate
  • 4
    Answer: top 10 products, revenue by region, monthly trend, customer lifetime value, anomalous orders anomaly
  • 5
    Write output as Parquet, partitioned by date
  • 6
    Push to GitHub with README explaining what you built and why each decision was made
Week 2 · Days 8–14 Hadoop Ecosystem + SQL Mastery Core infrastructure
DAY 08 Learn
HDFS Hands-On with Docker
Run a single-node Hadoop cluster using Docker. Put files into HDFS using hdfs dfs commands. Read them back. See the block structure. Check NameNode web UI.
Output: Working Docker Hadoop setup + 10 HDFS commands practiced
DAY 09 Anomaly
HDFS — Simulate NameNode Pressure
Create 10,000 tiny files in HDFS. Check NameNode memory usage. Then merge them into fewer larger files. Experience firsthand why small file problem is a real production issue.
Output: Documented small file problem + solution approach
DAY 10 Learn
Hive — Tables, Partitions, Bucketing
Create an external Hive table on HDFS data. Add partitions by date. Run queries with and without partition pruning — measure the difference in data scanned. Then create a bucketed table.
Output: Hive script with partitioned + bucketed table examples
DAY 11 Anomaly
Schema Mismatch — The JAR Story Scenario
Simulate your real production incident: create a pipeline reading a table, then add a column to the source without updating the pipeline. Watch it fail. Debug the error. Fix it. Document root cause.
Output: Incident report + fix — your real story, now reproducible
DAY 12 Practice
Advanced SQL — 10 Hard Problems
Solve 10 window function + CTE problems on LeetCode SQL or HackerRank. Focus: DENSE_RANK, LAG/LEAD, running totals, conditional aggregation. Time yourself — 10 min per problem max.
Output: 10 solved problems, screenshot your scores
DAY 13 Learn
Delta Lake on Databricks Community
Log into Databricks Community (free). Create a Delta table. Do an upsert (MERGE). Use time travel to query a previous version. Break schema enforcement intentionally — see the error.
Output: Databricks notebook with Delta Lake CRUD + time travel
DAY 14 Build
Mini Project — Hive + Delta Pipeline
Build: raw data lands in HDFS → Hive external table created → data transformed → written to Delta Lake → MERGE upsert handles duplicates. Push to GitHub.
Output: GitHub repo — Project 2 of 3 complete ✅
Project 2 Hive + Delta Lake Ingestion Pipeline

Simulate a real enterprise data ingestion pattern: raw files land in HDFS, a Hive external table exposes them, PySpark transforms the data, and the result is written to a Delta Lake table with upsert logic. Includes intentional anomalies you debug and fix.

  • 1
    Generate synthetic raw data (Python faker library) — employee/transaction records
  • 2
    Land in HDFS, create external Hive table with partitioning
  • 3
    Introduce duplicate records + late-arriving data anomaly
  • 4
    Transform with PySpark — clean, deduplicate, enrich
  • 5
    Write to Delta Lake with MERGE (upsert) to handle duplicates
  • 6
    Use Delta time travel to show state before and after MERGE anomaly
  • 7

Week 3 · Days 15–21 ETL Pipelines + Data Modeling + NoSQL Building real pipelines
DAY 15 Learn
ETL vs ELT + Medallion Architecture
Build a 3-layer Medallion pipeline: Bronze (raw), Silver (cleaned), Gold (aggregated). Use real data. Each layer writes to a separate folder/Delta table. This is standard at every company using Databricks.
Output: 3-layer Medallion pipeline in PySpark/Databricks
DAY 16 Anomaly
Late-Arriving Data Problem
Build a daily pipeline. Then send data for yesterday with today's run. How do you handle it? Implement watermarking logic. See how Delta MERGE solves this problem cleanly.
Output: Pipeline with late-arrival handling logic documented
DAY 17 Learn
Data Modeling — Star Schema
Design a star schema for an e-commerce dataset: fact_orders, dim_customer, dim_product, dim_date. Create tables in Hive/DBeaver. Run analytical queries. Understand why data warehouses use this design.
Output: ERD diagram + working star schema with 5 analytical queries
DAY 18 Anomaly
Data Skew — When One Partition Kills the Job
Create a dataset where 80% of rows have the same key. Run a groupBy and watch Spark UI — one task takes 10x longer. Apply salting technique to fix skew. See the job balance.
Output: Skew before/after comparison + salting solution documented
DAY 19 Learn
NoSQL — HBase + MongoDB Basics
Run MongoDB in Docker. Insert documents, query with filters, understand when NoSQL beats SQL (unstructured data, variable schema, high write throughput). Learn HBase column-family concept conceptually.
Output: MongoDB CRUD script + written comparison of when to use NoSQL vs SQL
DAY 20 Practice
Mock Interview — Concept + SQL Combined
Do a self-timed mock: 5 concept questions (write answers, not just think them) + 3 SQL problems from LeetCode Hard category. Grade yourself honestly. Note weak spots.
Output: Written answers + honest self-grade + weak spot list
DAY 21 Build
Mini Project — Full ETL with Anomaly Handling
Build a pipeline that handles: nulls, duplicates, late data, schema drift, and data skew — all in one run. Medallion architecture. Document every anomaly and how you handled it.
Output: Most complete project yet — push to GitHub ✅

⚠️ Honest Expectations — Read Before You Start

  • Day 30 makes you interview-ready for entry-level Data Engineer conversations — not senior roles. That takes another 30-60 days of depth work.
  • Missing even 3-4 days breaks the compounding. If life happens, make up days on weekends — don't skip and move forward.
  • Projects matter more than topics studied. An interviewer who sees 3 working GitHub projects will forgive gaps in theoretical knowledge. Someone with no projects but perfect notes gets rejected.
  • 2 hours/day means 2 focused hours — not 2 hours with phone nearby. Distracted hours don't count toward this plan.
  • The anomaly days are not optional. They're the difference between "I studied this" and "I've debugged this" — the latter is what gets you hired.
